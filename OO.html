Diagrama de Clases

Clase: Producto
Atributos:
nombre: Nombre del helado.
descripcion: una breve descripcion de el producto
precio: Costo del helado.
sabor: Sabor del helado.
Cant_Disponible: Cantidad disponible en inventario.
Descripción: Representa cada tipo de helado que se ofrece en la tienda. Cada producto tiene un nombre, un precio y un sabor, y también se gestiona el stock para asegurar que los pedidos puedan ser atendidos.

Clase: Inventario
Atributos:
producto: numero de productos
Descripción: Gestiona el conjunto de productos disponibles para la venta. En este caso, el inventario contiene varios productos, y su relación con la clase Producto es una agregación. Esto significa que el inventario puede existir sin productos (por ejemplo, un inventario vacío), y los productos pueden ser independientes del inventario (pueden existir en otras tiendas).

Clase: Cliente
Atributos:
nombre: Nombre del cliente.
email: Correo electrónico del cliente.
direccion: Dirección de envío del cliente.
telefono: Numero al cual solicitar
Descripción: Representa a los usuarios que interactúan con el sistema para realizar compras. La clase Cliente está relacionada con la clase Pedido mediante una asignación, ya que un cliente puede realizar uno o más pedidos, pero cada pedido pertenece a un único cliente.

Clase: Pedido
Atributos:
idPedido: identificador del pedido
cliente: el cliente que hizo e pedido
listaDeProductos: lista de productos que hizo el cliente
total: Total del costo del pedido.
Descripción: Representa el conjunto de productos que un cliente desea comprar en una transacción. La relación entre Pedido y Pago es de composición, ya que un pedido no tiene sentido sin un pago asociado. Si se cancela el pedido, el pago también se cancela, lo que indica que la existencia de un Pedido depende de un Pago.

Clase: Pago
Atributos:
metodoPago: Método de pago (tarjeta de crédito, PayPal, etc.).
total: Monto total del pago realizado.
fechaPago: Fecha en que se realizó el pago.
Descripción: Representa la transacción de pago por un pedido. Está directamente relacionado con el Pedido, ya que cada pedido requiere un pago para ser completado. Esta relación es de composición porque el pago no puede existir sin un pedido asociado.

Resumen de las relaciones:
Agregación (Inventario ↔ Producto):
Los productos pueden existir independientemente del inventario. El inventario puede ser vacío, pero los productos son entidades que pueden vivir en otros contextos o en otras tiendas.
Composición (Pedido ↔ Pago):
Un pedido no tiene sentido sin un pago asociado. Si se elimina un pedido, se debe eliminar también el pago relacionado, reflejando que el pago depende completamente del pedido.
Asignación (Cliente ↔ Pedido):
Cada cliente puede realizar múltiples pedidos, pero cada pedido pertenece a un solo cliente, lo que establece una relación directa y uno a muchos.
Asignación (Pedido ↔ PedidoProducto):
Cada pedido puede incluir múltiples productos (representados en la clase PedidoProducto), y cada producto puede estar en diferentes pedidos, lo que permite una relación muchos a muchos.


Diagrama de Casos de Uso

Descripción del Caso de Uso: Gestionar pedidos de helados
 Este caso de uso detalla la interacción entre el cliente, el administrador y el sistema para completar el proceso de compra y gestión de pedidos en una tienda de helados en línea. El cliente selecciona productos, realiza el pago y confirma el pedido, mientras que el administrador supervisa y gestiona los pedidos para su preparación o envío. El sistema actúa como intermediario, registrando los pedidos, verificando los datos y notificando tanto al cliente como al administrador sobre el estado del proceso.

Diagrama de Actividades

Descripción del Diagrama de Actividades: Gestionar pedidos de helados
El diagrama de actividades representa el flujo de acciones necesarias para completar el proceso de gestión de pedidos. Este inicia con el cliente navegando por la tienda y seleccionando helados. Luego, el cliente confirma el pedido tras ingresar sus datos de pago y envío, lo que desencadena validaciones por parte del sistema. Si los datos son válidos, el sistema registra el pedido y lo envía al administrador, quien revisa su disponibilidad en el inventario. Según el resultado, el administrador puede aceptar o rechazar el pedido. Finalmente, el cliente recibe una notificación sobre el estado actualizado, concluyendo el proceso. El diagrama incluye decisiones clave, como validaciones de pago e inventario, y muestra las interacciones entre los actores y el sistema de manera secuencial y lógica.
